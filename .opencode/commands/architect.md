## Phase 0: Baseline State Initialization
Before interacting with the user, you must establish the project's current state.
1. Use your file-reading tool to open and read the contents of `sdlc/planning/PRD.md`.
2. Absorb the functional requirements, scope, and product goals defined during the planning stage. 
3. Do **NOT** re-interview the user on high-level feature requirements that are already documented in the PRD.

## Phase 1: Technical Architecture Discovery (Dynamic Questioning)
Your objective is to interview the user to gather the technical, structural, and infrastructure requirements necessary to form a system architecture design.

* **Goal:** Collect sufficient technical details to map out system components, data flow, tech stack choices, and deployment strategies.
* **Execution:** Ask **one or two targeted technical questions at a time**. Do not overwhelm the user.
* **Scope:** Focus strictly on architectural choices:
  * **Tech Stack:** Frontend/backend frameworks, databases, cache layers, external APIs, and programming languages.
  * **Data Modeling:** High-level entities, state management strategies, or data persistence constraints.
  * **Infrastructure & Security:** Cloud providers, local deployment vs. hosted, authentication strategies, and data privacy needs.

## Phase 2: The Gate (Validation)
Iterate on the technical options until you have a cohesive architectural blueprint. Do **NOT** proceed to the bash commands in Phase 3 until you explicitly meet this condition:
* You have a clear architectural direction **AND** the user explicitly indicates they are ready to process it (e.g., "I'm ready", "Run the script", "Generate the architecture").

## Phase 3: Context Export & Script Execution
Once the gate in Phase 2 is passed, execute the following steps exactly using your Bash tool:

1. Ensure the architecture discussions directory exists:
   ```bash
   mkdir -p discussions/architecture
   ```
2. Save the full technical conversation context and decisions to a timestamped file:
   ```bash
   CONTEXT_FILE="discussions/architecture/$(date +%Y%m%d_%H%M%S)-architecture-context.md"
   cat > "$CONTEXT_FILE" << 'EOF'
   [Insert complete chronological technical conversation history, selected stack, data flows, and architectural decisions here]
   EOF
   ```
3. Execute the LangGraph pipeline execution script for the architecture stage:
   ```bash
   python3 .scripts/sdlc_harness.py --stage architecture --context "$CONTEXT_FILE"
   ```

## Phase 4. Output Synthesis & Handover
Read the stdout/stderr printed by the Python script.
1. **Summarize Changes:** Outline the technical state changes, system diagrams generated (if any), or structural blueprints returned by the script.
2. **Clear Context:** From this point forward, treat the prior technical brainstorming conversation as deprecated noise to optimize current memory constraints.
3. **Pivot Context:** Read the output architectural specifications or system design documents generated by your script (e.g., sdlc/architecture/architecture_spec.md) using your file-reading tool to prepare your internal state for the next step.
4. **Trigger Next Stage:** Prompt the user that the architecture stage is complete and instruct them on how to move to the UI-Design stage.
_Example message to user:_
"Architecture specifications exported and state updated. To begin the Requirements stage with the clean architectural and PRD baseline, run: /requirements"
5. CRITICAL: Hold the line here. Do NOT automatically begin generating code files, components, or UI mockups yourself. Wait for human review and the next explicit command invocation.
