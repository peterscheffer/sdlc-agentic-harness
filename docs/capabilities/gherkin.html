Gherkin Specifications & Requirements Engineering

The requirements stage produces both a structured REQUIREMENTS.md document and Gherkin .feature files that serve as executable specifications. The testing stage then uses AI inference to verify that all Gherkin scenarios are implemented in the source code.

Requirements Document

sdlc/requirements/REQUIREMENTS.md contains three structured tables:

SectionColumnsPurpose
Functional RequirementsID, Description, Priority (High/Medium/Low)What the system must do
Non-Functional RequirementsID, DescriptionPerformance, security, scalability constraints
Behavioural RequirementsID, Scenario, Expected BehaviourHow the system behaves in specific situations

Gherkin Feature Files

The LLM generates one .feature file per feature area, written in standard Gherkin syntax:

Feature: User Authentication

  Scenario: Successful login
    Given the user is on the login page
    When they enter valid credentials
    Then they are redirected to the dashboard

  Scenario: Failed login with invalid password
    Given the user is on the login page
    When they enter an invalid password
    Then an error message is displayed

Generated feature files are written to sdlc/requirements/ with kebab-case filenames (e.g., user-authentication.feature).

AI-Based Gherkin Compliance Verification

The testing stage runs an AI inference compliance check that feeds Gherkin feature files and the implemented source code to an LLM. The LLM determines whether all scenarios are adequately addressed in the code.

The verification prompt structure (from nodes/testing.py):

prompt_parts = [
    "## Gherkin Specifications\n" + gherkin_specs,
    "\n\n## Source Code\n" + source_files,
    "\n\n## Instructions\n",
    "Review the Gherkin feature files and the source code implementation. ",
    "Determine if ALL the Gherkin scenarios are adequately addressed in the code. ",
    "Respond with exactly one of:\n",
    "- 'GHERKIN_COMPLIANCE: pass' if all scenarios are implemented\n",
    "- 'GHERKIN_COMPLIANCE: fail' with a brief explanation of what is missing",
]

The LLM's response is parsed programmatically:

  • GHERKIN_COMPLIANCE: pass → gate passes
  • GHERKIN_COMPLIANCE: fail → gate fails, stage status set to failed
  • Unexpected response → gate fails

The compliance result is appended to TEST_REPORT.md:

## Gherkin Compliance
**Status:** ✓ PASSED
**Message:** All Gherkin scenarios are implemented

Future Enhancement: Executable Gherkin

As noted in docs/SPEC_GAPS.md, the current AI-based verification is a stepping stone toward fully executable Gherkin. The highest-leverage enhancement would be integrating pytest-bdd to run .feature files as real passing/failing tests, replacing the LLM compliance guess with actual test results.
