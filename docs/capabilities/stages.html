8-Stage SDLC Pipeline

The harness enforces a sequential, gate-verified pipeline from intent parsing through GitHub PR submission. Each stage produces auditable artefacts and halts until the developer issues the next command. Stages can be skipped (ui-design) or re-entered (coding from review).

1: planning

Required Output: sdlc/planning/PRD.md

Parses the developer's intent string and produces a structured Product Requirements Document. Uses a configurable LLM model to fill in a PRD template with sections including Summary, Goals, Non-Goals, Tasks, Acceptance Criteria, and Affected Files. Validates that the output contains all required section headings and at least one checkbox task before advancing.

Gates: PRD.md exists, PRD schema valid, tasks defined

2: ui-design (Optional)

Optional — auto-skipped Output: sdlc/ui-design/DESIGN.md

Generates a UI design specification conforming to Google Stitch DESIGN.md format. Automatically skipped if the PRD contains no UI-relevant keywords (component, page, view, screen, UI, frontend, form, modal, layout). The developer can force-run or force-skip this stage.

Gates: DESIGN.md exists, schema valid (## Overview + ## Components or ## Screens)

3: architecture

Required Output: sdlc/architecture/ARCH.md

Produces an Architecture Decision Record (ARCH.md) that includes an overview, target files table, design decisions, PRINCIPLES compliance section, and risks. Validates the architecture against every rule defined in PRINCIPLES.md — violations with severity:error block the stage.

Gates: ARCH.md exists, schema valid, zero error-severity PRINCIPLES violations

4: requirements

Required Output: sdlc/requirements/REQUIREMENTS.md + .feature files

Generates a structured requirements document with functional, non-functional, and behavioural requirements tables. Also produces Gherkin .feature files (one per feature area) that serve as executable specifications for the testing stage.

Gates: REQUIREMENTS.md exists, schema valid, at least one .feature file exists

5: coding

Required Output: Modified source files + ITERATIONS.md

Iteratively generates and corrects source code (the "Coding Loop") until all gate checks pass or max iterations are reached. Each iteration starts with a fresh LLM context to prevent context rot. See Coding Loop for details.

Gates: Linter passes, build passes, all target files exist

6: testing

Required Output: sdlc/testing/TEST_REPORT.md

Executes the project's configured test command, captures output and exit code, parses coverage percentages, and writes a structured test report. Also runs an AI-based Gherkin compliance check to verify that all Gherkin scenarios are implemented in source code.

Gates: Tests pass, TEST_REPORT.md exists, report not empty, Gherkin compliance, optional coverage threshold

7: review

Required Output: sdlc/review/REVIEW.md

Performs a structured self-review using the git diff, PRD, architecture, and test report as context. Produces a REVIEW.md with change summary, PRD alignment, PRINCIPLES compliance, test evidence, and a PASS/FAIL recommendation. A FAIL recommendation blocks PR submission unless overridden with --force.

Gates: REVIEW.md exists, recommendation field present (PASS or FAIL)

8: pr

Required Output: GitHub PR URL

Commits all SDLC artefacts to the current branch, constructs a structured PR body (summary, changes, test evidence, review outcome, artefact links), and submits via gh CLI. Captures the PR URL in pipeline state.

Gates: gh CLI authenticated, not on base branch, PR created successfully

Cross-Stage Workflows

The pipeline supports non-linear navigation:

  • Re-enter planning: A new intent on an existing pipeline creates a fresh pipeline
  • Re-enter coding from review: If review produces FAIL, return to the coding loop with review context
  • Rerun testing: Re-execute after fixing test failures without re-running coding
  • PRD auto-update: After each completed stage, an LLM evaluates whether the PRD should be updated with new information
