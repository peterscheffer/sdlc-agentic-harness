Dual Verification Methods

The harness supports two verification modalities, applied with a clear priority: code-based verification first, AI inference second. This ensures that wherever possible, verification is deterministic, fast, and repeatable.

Primary: Code-Based Verification

Code-based checks are preferred because they are deterministic, auditable, and produce no LLM cost. These run as shell commands or file system checks:

  • Lint commands — validate code style and static analysis (e.g., ruff, eslint)
  • Build commands — verify compilation or bundling succeeds
  • Test commands — execute the project's test suite, capture exit code and output
  • Coverage parsers — extract and validate code coverage percentages from test output
  • File existence checks — confirm that architecture-targeted files were created on disk
  • Schema validation — check artefact files for required section headings

Each command is configured in sdlc.config.json under commands:

{
  "commands": {
    "lint": "ruff check .",
    "build": "python -m compileall .",
    "test": "python -m pytest tests/ -q",
    "coverage_report": null
  }
}

Secondary: AI Inference Verification

AI inference is used where code-based verification is impractical or impossible. The harness invokes an LLM to evaluate compliance:

  • Gherkin compliance — feeds Gherkin .feature files and implemented source code to an LLM, which determines whether all scenarios are adequately addressed. The LLM must respond with exactly GHERKIN_COMPLIANCE: pass or GHERKIN_COMPLIANCE: fail
  • PRD alignment — the review stage uses an LLM to verify the git diff satisfies every PRD task and acceptance criterion
  • PRINCIPLES compliance — LLM evaluates the architecture document against project-specific principle rules defined in PRINCIPLES.md
  • PRD auto-update — after each stage, an LLM determines whether newly discovered information should be incorporated into the PRD
# AI verification prompt pattern from testing.py:
"Determine if ALL the Gherkin scenarios are adequately
 addressed in the code. Respond with exactly one of:
 - 'GHERKIN_COMPLIANCE: pass'
 - 'GHERKIN_COMPLIANCE: fail'"

Why This Ordering?

The code-first approach ensures that verification is deterministic (same input always produces same result), fast (milliseconds, not seconds), and zero-cost (no LLM inference tokens). AI inference is reserved for tasks that require semantic understanding — interpreting natural language requirements, evaluating architectural compliance, or verifying behavioural specifications against implementation.
