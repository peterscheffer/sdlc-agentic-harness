# Spec-to-Verification Gaps by Stage

## Planning (PRD)

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Task checklist | All tasks are well-scoped and achievable | Currently manual. Could add a pre-gate LLM check: *"For each task, is it specific, measurable, and testable?"* |
| Acceptance criteria ACID test | AC are distinct, verifiable, unambiguous | LLM gate check before saving PRD — catch vague ACs early |
| PRD consistency | No contradictory goals or requirements | Automated cross-section consistency check |

## UI Design (missing the most)

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Component inventory | Every UI component in DESIGN.md exists in code | Gate reads DESIGN.md, globs source for matching component names/directories |
| Interaction states | All declared states (loading, empty, error) are handled | Check that each state has a corresponding code path, CSS class, or conditional |
| Layout assertions | Page layout matches spec | Visual regression diff (Playwright screenshot comparison against a mock) |
| Accessibility spec | a11y requirements from design are met | Automated aXe/Pa11y check gated on design's stated a11y targets |

## Architecture

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Target files list | All specified files exist AND no extra untracked files | Already have `target_files_exist`. Could add `target_files_only` gate (no stray files) |
| Dependency rules | Modules only import from allowed layers | Architecture fitness function via `pytest-arch` or custom `check_imports()` gate |
| API contract integrity | Implementation matches OpenAPI spec if generated | `prance validate openapi.json` or schemathesis fuzz-testing |
| ADR compliance | Decisions in ADRs are followed in code | Per-ADR gate check (customizable per decision) |

## Requirements (already strongest)

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Executable Gherkin | `.feature` scenarios pass as automated tests | `pytest-bdd` runner — replaces the LLM compliance guess with actual pass/fail |
| Requirement → scenario traceability | Every requirement maps to ≥1 Gherkin scenario | Cross-reference table: no orphan requirements, no untested scenarios |
| Property-based invariants | Edge cases beyond example scenarios | Hypothesis tests generated from behavioural requirements tables |

## Coding

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Spec-first generation | Code is generated *from* the spec, not independently | Reverse the Coding loop: feed Gherkin scenarios as the prompt, code must satisfy them |
| Architecture alignment | Generated code structure matches ARCH.md's target files and design decisions | Cross-check imports, class names, file paths against architect's declaration |
| Cyclomatic complexity budget | Code stays within spec'd complexity limits | Gate on `radon cc` or `lizard` output against thresholds set in architecture |
| Test coverage for each file | Every target file has corresponding tests | Gate on per-file coverage, not just global percentage |

## Testing

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Executable Gherkin | Feature files become test suites | `pytest-bdd` — this is the single biggest gap-closer in the pipeline |
| Mutation score | Tests actually catch real bugs (not just passing against known code) | `mutmut` or `pytest-mutagen` — kills mutants; fail gate if below threshold |
| Gherkin → test coverage | Every scenario has a corresponding passing test | Traceability matrix generated by test framework |
| Performance budget | Performance thresholds from REQUIREMENTS.md are met | `pytest-benchmark` with thresholds |

## Review

| Spec | What it verifies | Approach |
|------|------------------|----------|
| AC coverage | All acceptance criteria are satisfied by the diff | LLM check (you have this) OR a structured checklist: each AC ticked off against code evidence |
| Spec diff | Only files that the spec said would change are changed | `git diff --name-only` vs ARCH.md's target files — flag unexpected files |
| Conventional commit | PR title/body follows a convention | `commitlint` in PR gate |

## PR

| Spec | What it verifies | Approach |
|------|------------------|----------|
| Artefact completeness | All expected SDLC artefacts are committed | Check the full artefact manifest against `git ls-files` |
| PR template compliance | Body covers all required sections | Gate on presence of key sections/sentences in `gh pr view --json body` |
| Link-to-requirements | PR references the originating spec | Check PR body contains pipeline ID or links to artefacts |

## Highest-leverage additions

1. **Executable Gherkin** (Testing stage) — `.feature` files become real passing/failing tests via `pytest-bdd`. Closes the spec-to-verification loop entirely.

2. **Architecture fitness functions** (Coding gates) — `pytest-arch` or similar enforces that code follows the architecture's dependency and structure decisions.

3. **Component inventory gate** (Coding/Review) — verifies every UI component declared in DESIGN.md exists in source, with all declared states handled.

Here's the tooling landscape:
Off-the-shelf tools that can verify certain rule categories:
Rule category	Example rule	Existing tool
Import/layering	"Module X must not import module Y"	pytest-arch
Type annotations	"All public functions have type hints"	mypy --strict or pyright
Secrets in code	"No hardcoded API keys"	gitleaks / trufflehog / detect-secrets
Naming conventions	"Functions use snake_case"	pylint / ruff with custom rules
File/structural rules	"Every module has __init__.py"	Custom script or pytest glob check
But most rules need custom code:
PRINCIPLES.md rules like:
- "Pipeline follows strict sequential SDLC with LangGraph state machine"
- "State survives terminal restarts"
- "LLM calls are logged for auditability"
These are semantic/institutional rules — they require custom check functions that understand what "correct" looks like for your pipeline. No off-the-shelf tool can verify them generically.
The pattern that would close the gap:
Add an optional check_type field to each PRINCIPLES.md rule:
### QUAL-001
description: All public functions have type annotations.
severity: warning
check_type: mypy                   # ← maps to existing tool
check: All public functions have type hints.
### ARCH-003
description: No direct DB access from view layer
severity: error
check_type: pytest-arch            # ← maps to existing tool
check: Views must not import database modules.
### CUSTOM-001
description: LLM calls are logged for auditability
severity: error
check_type: custom                 # ← maps to custom check script
check: sdlc/gates/checks/llm_logging.py
Then the pipeline:
1. Architecture stage: validates ARCH.md declares compliance (current behavior)
2. Coding gates: runs check_type tools against actual code
3. Review stage: runs remaining checks that need the full diff context
