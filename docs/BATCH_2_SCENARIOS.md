## Feature 5: Coding Stage – Iterative Code Generation with Ralph Loop

```gherkin
Feature: Coding Stage – Iterative Code Generation with Context Isolation
  As the SDLC pipeline
  I want to iteratively generate code until the build is clean
  So that code changes pass linter, build, and file existence checks

  Background:
    Given sdlc/architecture/ARCH.md has been generated and passed gate checks
    And the project has configured linter and build commands in sdlc.config.json
    And the coding model is available
    And the current working directory is the project root

  Scenario: Coding stage begins with fresh LLM context on iteration 1
    Given I have run "/sdlc coding"
    When the coding loop subgraph executes iteration 1
    Then the LLM is called with only:
      | context |
      | sdlc/architecture/ARCH.md |
      | Current target file states (read from disk) |
      | No prior conversation history |
    And the LLM generates initial code for target files

  Scenario: Iteration 1: Generate code for target files listed in ARCH.md
    Given the coding loop has started iteration 1
    And ARCH.md specifies target files: src/auth.ts, src/middleware.ts
    When the LLM executes
    Then it generates or modifies code for those files
    And the files are written to disk in their specified locations

  Scenario: Iteration 1: Run gate checks after code generation
    Given code has been generated by the LLM
    When gate checks execute
    Then the system runs these commands in sequence:
      | command |
      | <linter command from sdlc.config.json> |
      | <build command from sdlc.config.json> |
      | File existence check for all target files |
    And each command's exit code is captured

  Scenario: Iteration 1: Linter gate check passes
    Given the generated code is syntactically correct
    When the linter command executes
    Then it exits with code 0
    And the gate check "linter_passed" = true

  Scenario: Iteration 1: Build gate check passes
    Given the generated code compiles without errors
    When the build command executes
    Then it exits with code 0
    And the gate check "build_passed" = true

  Scenario: Iteration 1: File existence gate check passes
    Given ARCH.md lists target files: src/auth.ts, src/middleware.ts
    When the file existence check runs
    Then both files exist on disk
    And the gate check "target_files_exist" = true

  Scenario: All iteration 1 gate checks pass – coding stage succeeds
    Given all three gate checks pass in iteration 1:
      | check |
      | linter_passed |
      | build_passed |
      | target_files_exist |
    When gate checks complete
    Then the coding loop exits successfully
    And the stage marks as complete
    And .sdlc_state.json.stages.coding.status = "complete"
    And .sdlc_state.json.stages.coding.iterations = 1

  Scenario: Iteration 1 gate check fails – proceed to iteration 2
    Given the linter command exits with non-zero code
    And the linter output is: "TypeScript error on line 12: Expected type annotation"
    When gate checks complete
    Then the stage does NOT advance
    Instead, the loop continues to iteration 2
    And the loop state is prepared for the next iteration

  Scenario: Clear context between iterations to prevent accumulation
    Given iteration 1 failed and iteration 2 is about to start
    When the LangGraph loop node transitions to iteration 2
    Then all prior conversation history and context is discarded
    And the LLM is called fresh with ONLY:
      | context |
      | sdlc/architecture/ARCH.md |
      | Current target file states (from disk) |
      | The specific gate failure reason from iteration 1 |
      | No prior attempts, no chat history |

  Scenario: Iteration 2: Receive failure reason as context
    Given iteration 1 failed with linter error: "TypeScript error on line 12"
    When iteration 2 LLM call is made
    Then the prompt includes verbatim:
      """
      Previous iteration failed. Fix this error and retry:
      TypeScript error on line 12: Expected type annotation
      
      Current state of target files is attached. Regenerate to fix this specific error.
      """

  Scenario: Iteration 2: Modify code to fix the failure
    Given the LLM receives the failure reason
    When the LLM executes
    Then it examines the current file state
    And modifies the code to address the specific error
    And writes the updated files to disk

  Scenario: Iteration 2: Run gate checks again
    Given modified code has been generated
    When gate checks execute
    Then linter, build, and file checks all run again
    And exit codes are captured
    And results are compared against the previous iteration

  Scenario: Iteration 2 gate checks pass – loop exits
    Given all gate checks pass in iteration 2
    When gate checks complete
    Then the coding loop exits
    And the stage marks as complete
    And .sdlc_state.json.stages.coding.iterations = 2
    And .sdlc_state.json.stages.coding.status = "complete"

  Scenario: Iteration 2 gate checks fail – proceed to iteration 3
    Given the build command still fails in iteration 2
    When gate checks complete
    Then the loop continues to iteration 3
    And the iteration counter increments

  Scenario: Multiple iterations with different failures
    Given the coding loop max_iterations = 5
    When iteration 1 fails on linter, iteration 2 fails on build, iteration 3 passes
    Then the loop runs 3 times total
    And .sdlc_state.json records all iterations
    And the stage succeeds on iteration 3

  Scenario: Reach max_iterations without passing
    Given max_iterations = 5 in sdlc.config.json
    And iterations 1-5 all fail gate checks
    When iteration 5 completes
    Then the loop exits with status "failed"
    And the stage fails with error:
      """
      Coding stage failed: Maximum iterations (5) reached without passing gate checks.
      See sdlc/coding/ITERATIONS.md for full iteration history.
      Review the failure reason and retry: /sdlc coding
      """

  Scenario: Write iteration log on loop failure
    Given the coding loop has run 5 iterations and failed
    When the stage completes
    Then sdlc/coding/ITERATIONS.md is created
    And the file contains a table:
      | Iteration | Linter | Build | Files Exist | Failure Reason |
      | 1 | ✗ | - | ✓ | TypeScript error line 12 |
      | 2 | ✓ | ✗ | ✓ | Missing dependency @types/stripe |
      | 3 | ✓ | ✗ | ✓ | Tsconfig not found |
      | 4 | ✓ | ✓ | ✗ | src/new-file.ts not created |
      | 5 | ✗ | - | ✓ | Import cycle detected |

  Scenario: Iteration log is human-readable
    Given sdlc/coding/ITERATIONS.md exists
    When I examine the file
    Then it includes:
      - Timestamp of pipeline start
      - Max iterations configured
      - Each iteration's results and specific failure
      - Command outputs (linter/build errors) for context

  Scenario: Update .sdlc_state.json after coding stage success
    Given coding stage has passed all gate checks in iteration N
    When the stage completes
    Then .sdlc_state.json is updated:
      | field | value |
      | stages.coding.status | complete |
      | stages.coding.completed_at | <ISO 8601 timestamp> |
      | stages.coding.iterations | N |
      | stages.coding.gate_results.linter_passed | true |
      | stages.coding.gate_results.build_passed | true |
      | stages.coding.gate_results.target_files_exist | true |
      | completed_stages | [..., "coding"] |

  Scenario: Update .sdlc_state.json after coding stage failure
    Given coding stage has run max_iterations without success
    When the stage completes
    Then .sdlc_state.json is updated:
      | field | value |
      | stages.coding.status | failed |
      | stages.coding.iterations | <max_iterations> |
      | stages.coding.artefact | sdlc/coding/ITERATIONS.md |
      | current_stage | coding |

  Scenario: Developer can manually fix code and re-run
    Given coding stage has failed after max iterations
    And sdlc/coding/ITERATIONS.md shows the failures
    When the developer manually edits src/auth.ts to fix an issue
    Then they run "/sdlc coding" again
    And the loop restarts from iteration 1 with fresh context
    And the developer's manual fixes are read as the current file state

  Scenario: Display coding progress to developer during execution
    Given coding stage is running
    When LLM is generating code and gate checks are executing
    Then OpenCode displays (in real-time or on completion):
      """
      [coding] Iteration 1 of 5...
      [coding] - Generated: src/auth.ts, src/middleware.ts
      [coding] - Linter: ✗ TypeScript error line 12
      [coding] Iteration 2 of 5...
      [coding] - Modified: src/auth.ts
      [coding] - Linter: ✓
      [coding] - Build: ✓
      [coding] - File checks: ✓
      [coding] ✓ Coding stage passed in 2 iterations.
      """

  Scenario: Display coding failure to developer clearly
    Given coding stage has failed
    When the OpenCode command finishes
    Then the output displays:
      """
      [coding] ✗ Coding stage failed after 5 iterations.
      [coding] Last failure: Missing dependency @types/stripe
      [coding] See sdlc/coding/ITERATIONS.md for full details.
      Fix the issue manually or refine the PRD and retry: /sdlc coding
      """

  Scenario: Log all LLM calls in coding stage
    Given coding stage executes with 3 iterations
    When each LLM call completes
    Then sdlc/logs/coding_iteration_1_<timestamp>.log exists
    And sdlc/logs/coding_iteration_2_<timestamp>.log exists
    And sdlc/logs/coding_iteration_3_<timestamp>.log exists
    And each log includes: prompt (context provided), full response, token usage

  Scenario: No secrets in coding logs
    Given coding logs have been written
    When I examine sdlc/logs/coding_iteration_*
    Then no API keys, credentials, or sensitive data appear in the files
    And all logs are safe to commit to version control

  Scenario: Coding stage gate checks are hard criteria only
    Given the coding loop has generated code
    When gate checks execute
    Then ONLY these checks run:
      | check |
      | Linter command exit code = 0 |
      | Build command exit code = 0 |
      | All target files exist |
    And there is NO quality judgment by LLM (e.g., "is this good code?")
    And there is NO PRINCIPLES validation (that happens in review stage)
```

## Feature 6: Testing Stage

```gherkin
Feature: Testing Stage – Test Execution and Coverage Validation
  As the SDLC pipeline
  I want to run the project's test suite and validate coverage
  So that code changes are verified to work and meet quality thresholds

  Background:
    Given sdlc/architecture/ARCH.md exists (coding stage completed)
    And the project has a valid sdlc.config.json with test command configured
    And the project has a valid sdlc.config.json with coverage settings
    And the current working directory is the project root

  Scenario: Testing stage executes the configured test command
    Given I have run "/sdlc testing"
    When the testing stage executes
    Then it runs the command from sdlc.config.json.commands.test
    And the command is executed as a subprocess in the project root
    And stdout and stderr are captured

  Scenario: Test command exits successfully
    Given the test suite passes
    When the test command exits
    Then the exit code is 0
    And the gate check "tests_passed" = true

  Scenario: Capture test output and format into TEST_REPORT.md
    Given the test command has executed
    When the testing stage completes
    Then sdlc/testing/TEST_REPORT.md is created
    And the file includes:
      - Test command executed
      - Exit code
      - Summary of test results (pass/fail counts if available)
      - Stdout and stderr excerpts (last 500 chars)

  Scenario: Parse coverage from test output
    Given the test command outputs coverage data
    And sdlc.config.json.coverage.enabled = true
    When the testing stage executes
    Then the system extracts coverage percentage from the output
    And stores it in TEST_REPORT.md
    And records it in .sdlc_state.json.stages.testing.coverage_percent

  Scenario: Validate coverage meets minimum threshold
    Given sdlc.config.json.coverage.min_percentage = 80
    And the test suite reports 82% coverage
    When gate checks run
    Then the gate check "coverage_threshold_met" = true
    And the stage does not fail on coverage

  Scenario: Fail testing gate if coverage below minimum
    Given sdlc.config.json.coverage.min_percentage = 80
    And the test suite reports 72% coverage
    When gate checks run
    Then the gate check "coverage_threshold_met" = false
    And the stage fails with error:
      """
      Testing gate failed: Coverage 72% is below minimum 80%.
      Write additional tests and retry: /sdlc testing
      """

  Scenario: Skip coverage check if coverage is disabled
    Given sdlc.config.json.coverage.enabled = false
    When the testing stage executes
    Then no coverage threshold check runs
    And TEST_REPORT.md notes:
      """
      Coverage validation: Disabled
      """

  Scenario: Testing gate check: tests_passed
    Given the test command has executed
    When gate checks run
    Then the gate check "tests_passed" returns (exit_code = 0)
    And the result is recorded in .sdlc_state.json.stages.testing.gate_results

  Scenario: Testing gate check: test_report_exists
    Given the testing stage has completed
    When gate checks run
    Then the gate check "test_report_exists" returns (sdlc/testing/TEST_REPORT.md exists)

  Scenario: Testing gate check: report_not_empty
    Given sdlc/testing/TEST_REPORT.md exists
    When gate checks run
    Then the gate check "report_not_empty" returns (file size > 0 bytes)

  Scenario: Fail testing stage if test command exits non-zero
    Given the test suite fails
    When the test command exits with code 1
    Then the gate check "tests_passed" = false
    And the stage fails with error:
      """
      Testing stage failed: Test command exited with code 1.
      See sdlc/testing/TEST_REPORT.md for details.
      Fix failing tests and retry: /sdlc testing
      """
    And TEST_REPORT.md is written even on failure for debugging

  Scenario: Fail testing gate if TEST_REPORT.md is not created
    Given the testing stage crashes or times out
    When the stage completes
    Then the gate check "test_report_exists" = false
    And the stage fails with error:
      """
      Testing stage failed: TEST_REPORT.md was not generated.
      Retry with: /sdlc testing
      """

  Scenario: TEST_REPORT.md schema is valid
    Given sdlc/testing/TEST_REPORT.md exists
    When the gate check runs
    Then the file contains at minimum:
      - Timestamp of test execution
      - Test command executed
      - Exit code
      - Summary (pass/fail/skip counts if parsed)
      - Coverage percentage (if enabled)

  Scenario: Respect test command timeout
    Given sdlc.config.json.timeouts.command_seconds = 300
    And the test command is hanging
    When 300 seconds have elapsed
    Then the test subprocess is terminated
    And the stage fails with error:
      """
      Testing stage failed: Test command timed out after 300 seconds.
      """

  Scenario: Update .sdlc_state.json on successful testing completion
    Given testing stage has passed all gate checks
    When the stage completes
    Then .sdlc_state.json is updated:
      | field | value |
      | stages.testing.status | complete |
      | stages.testing.completed_at | <ISO 8601 timestamp> |
      | stages.testing.artefact | sdlc/testing/TEST_REPORT.md |
      | stages.testing.coverage_percent | <coverage % or null> |
      | stages.testing.gate_results.tests_passed | true |
      | stages.testing.gate_results.test_report_exists | true |
      | stages.testing.gate_results.report_not_empty | true |
      | stages.testing.gate_results.coverage_threshold_met | true or null |
      | completed_stages | [..., "testing"] |

  Scenario: Update .sdlc_state.json on testing failure
    Given testing stage has failed a gate check
    When the stage completes
    Then .sdlc_state.json is updated:
      | field | value |
      | stages.testing.status | failed |
      | stages.testing.gate_results.<check_name> | false |
      | current_stage | testing |

  Scenario: Display testing completion to developer
    Given testing stage has completed successfully
    When the OpenCode command finishes
    Then the output displays:
      """
      [testing] ✓ Tests executed: 145 passed, 0 failed
      [testing] ✓ Coverage: 84% (exceeds 80% minimum)
      [testing] ✓ Gate checks passed (4/4)
      See sdlc/testing/TEST_REPORT.md for details.
      To review the changes, run: /sdlc review
      """

  Scenario: Display testing failure to developer
    Given testing stage has failed
    When the OpenCode command finishes
    Then the output displays:
      """
      [testing] ✗ Tests failed: 3 failures
      [testing] ✗ Test command exited with code 1
      [testing] See sdlc/testing/TEST_REPORT.md for error details.
      Fix failing tests and retry: /sdlc testing
      """

  Scenario: Log test execution for auditability
    Given the testing stage has executed
    When the stage completes
    Then sdlc/logs/testing_<timestamp>.log is created
    And the log includes:
      - Timestamp of execution
      - Test command executed
      - Full stdout and stderr
      - Coverage data extracted (if applicable)
      - Exit code

  Scenario: No secrets in testing logs
    Given testing logs have been written
    When I examine sdlc/logs/testing_*
    Then no API keys, database credentials, or sensitive data appear
    And logs are safe to commit

  Scenario: Re-run testing stage without re-running coding
    Given the testing stage has completed
    When I modify a test file manually and run "/sdlc testing" again
    Then the coding stage is NOT re-run
    And only the test command executes
    And the existing generated source files are used as-is

  Scenario: Testing stage is independent of coverage tool choice
    Given the project uses a custom coverage tool or no coverage at all
    When sdlc.config.json.coverage.enabled = false
    Then the testing stage executes the test command
    And no coverage parsing is attempted
    And the stage succeeds based on exit code only
```

## Feature 7: Stage Continuation and Gating

```gherkin
Feature: Stage Gate Enforcement and Pipeline Progression
  As the SDLC pipeline
  I want to strictly enforce gate checks before allowing progression
  So that only verified stages can proceed to the next step

  Background:
    Given the project has completed planning through testing stages

  Scenario: Cannot advance to review if coding stage is not complete
    Given .sdlc_state.json shows stages.coding.status = "failed"
    When I run "/sdlc review"
    Then the command fails with error:
      """
      Error: Cannot advance to review. 
      Required stage 'coding' has not been completed.
      Retry coding stage: /sdlc coding
      """
    And no changes are made to .sdlc_state.json

  Scenario: Cannot advance to review if testing stage is not complete
    Given .sdlc_state.json shows stages.testing.status = "not_started"
    When I run "/sdlc review"
    Then the command fails with error:
      """
      Error: Cannot advance to review. 
      Required stage 'testing' has not been completed.
      Run testing stage: /sdlc testing
      """

  Scenario: Can advance to review only after all gate checks pass
    Given all of coding, testing, and review gate checks have passed
    When I run "/sdlc review"
    Then the review stage executes
    And .sdlc_state.json.current_stage transitions to "review"

  Scenario: Gating prevents skipping stages
    Given I have completed planning and architecture
    When I run "/sdlc testing"
    Then the command fails with error:
      """
      Error: Cannot skip to testing. 
      Expected next stage: coding
      """

  Scenario: Display gate check results to developer
    Given a stage has completed
    When all gate checks pass
    Then OpenCode displays:
      """
      [<stage>] ✓ Gate checks passed (<pass_count>/<total_count>)
      """
    And if any gate checks fail:
      """
      [<stage>] ✗ Gate check failed: <check_name>
      """
```

## Feature 8: Error Handling and Retries

```gherkin
Feature: Error Handling, Retries, and Recovery
  As the SDLC pipeline
  I want to handle failures gracefully and allow retries
  So that transient errors don't break the pipeline irreversibly

  Background:
    Given the project has valid configuration

  Scenario: Retry LLM call up to 3 times on failure
    Given an LLM call fails (timeout, API error, etc.)
    When the LangGraph node executes
    Then it retries automatically:
      | attempt | delay |
      | 1 | immediate |
      | 2 | 2 seconds |
      | 3 | 4 seconds |
    And if all 3 retries fail, the stage fails with error

  Scenario: Fail stage with clear error message after all retries
    Given 3 LLM call retries have all failed
    When the stage completes
    Then the error message includes:
      """
      <Stage> stage failed: LLM call failed after 3 retries.
      Last error: <error details>
      Retry the stage: /sdlc <stage>
      """

  Scenario: Timeout on command execution
    Given sdlc.config.json.timeouts.command_seconds = 300
    And a shell command (linter, build, test) is hanging
    When 300 seconds elapse
    Then the subprocess is forcibly terminated
    And the gate check fails with error:
      """
      Command timed out after 300 seconds: <command>
      """

  Scenario: Timeout on LLM call
    Given sdlc.config.json.timeouts.llm_call_seconds = 120
    And an LLM call is taking too long
    When 120 seconds elapse
    Then the call is cancelled
    And retry logic is triggered (up to 3 times)

  Scenario: Recover from corrupted .sdlc_state.json
    Given .sdlc_state.json is malformed JSON
    When a stage tries to read it
    Then the system detects the error
    And displays:
      """
      Error: .sdlc_state.json is corrupted.
      Clear it with: /sdlc reset
      """

  Scenario: Handle missing sdlc.config.json
    Given no sdlc.config.json file exists in the project root
    When a stage tries to execute
    Then the pipeline fails immediately with error:
      """
      Error: sdlc.config.json is missing.
      Please create it with the required configuration.
      Reference: CONTRIBUTING.md
      """

  Scenario: Handle invalid sdlc.config.json schema
    Given sdlc.config.json exists but has missing required fields
    When a stage tries to execute
    Then Pydantic validation fails
    And the error displays:
      """
      Error: sdlc.config.json validation failed:
      - default_model: required field (missing)
      See schema reference: <docs link>
      """

  Scenario: Handle missing linter command in config
    Given sdlc.config.json.commands.lint is empty or missing
    When the coding stage runs and linter gate check is due
    Then the gate check fails with error:
      """
      Error: Linter command not configured in sdlc.config.json.
      Add: commands.lint: "<your_linter_command>"
      """

  Scenario: Handle missing test command in config
    Given sdlc.config.json.commands.test is empty or missing
    When the testing stage runs
    Then the stage fails with error:
      """
      Error: Test command not configured in sdlc.config.json.
      Add: commands.test: "<your_test_command>"
      """
```
