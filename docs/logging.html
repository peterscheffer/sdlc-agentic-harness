LLM Interaction Logging

Every LLM invocation by the harness is logged to sdlc/logs/ in structured JSON format. This provides a complete audit trail for every prompt, response, and model decision across all pipeline stages.

Log Structure

Each log file is named <stage>_<timestamp>.log (or <stage>_iteration_<N>_<timestamp>.log for coding iterations) and contains:

{
  "timestamp": "2026-06-15T10:30:00.000000+00:00",
  "stage": "planning",
  "model": "qwen3.6:35b-a3b",
  "provider": "ollama",
  "prompt_tokens": 1250,
  "completion_tokens": 890,
  "system_prompt": "You are an expert product requirements...",
  "prompt": "## Conversation Context\n\nFeature: Add JWT...",
  "response": "# Product Requirements Document\n\n## Summary...",
  "iteration": 1,
  "has_conversation_context": true
}

Automatic Secrets Redaction

After every log write, the harness runs a secrets redaction pass against the log file. This ensures no sensitive information leaks into the audit trail:

# Redaction patterns (from utils/llm.py):
patterns = [
  r'(?i)(api[-_]?key|apikey|secret|token|password|auth)[=:]\s*\S+',
  r'(?i)(authorization|bearer)\s+\S+',
  r'(?i)(sk-[a-zA-Z0-9]{20,})',
]

# Result: "api_key=sk-abc123..." → "api_key= [REDACTED]"

This covers: API keys Secrets Tokens Passwords Bearer tokens OpenAI-style sk- keys

Retry Tracking

When an LLM call fails, retry attempts are recorded in the log entry:

{
  "retries": [
    { "attempt": 1, "error": "Connection refused" },
    { "attempt": 2, "error": "Connection refused" }
  ],
  "error": "LLM call failed after 3 retries. Last error: ..."
}

Mock LLM Mode

For testing without real LLM calls, set the SDLC_USE_MOCK_LLM environment variable:

export SDLC_USE_MOCK_LLM=1
python3 .scripts/sdlc_harness.py --stage planning --feature "add auth"

This returns deterministic mock responses for every stage, logs them with a (mock) indicator, and enables full pipeline testing without any LLM dependency. The test suite uses this mode exclusively.

Per-stage model selection allows using different models for different cognitive tasks — a smaller model for planning, a larger model for coding:

{
  "stages": {
    "planning":     { "model": "gemma4:latest" },
    "architecture": { "model": "gemma4:latest" },
    "coding":       { "model": "qwen3.6:35b-a3b", "max_iterations": 5 },
    "review":       { "model": "gemma4:31b" }
  }
}

Benefits

  • Auditability: Every AI decision is recorded and reviewable
  • Debugging: Failed stages can be traced back to the exact prompt and response
  • Cost tracking: Token counts enable cost analysis per stage and per pipeline run
  • Security: Secrets are automatically redacted before logs are committed
  • Reproducibility: Mock mode allows deterministic replay of any pipeline run
